# Taste Skill

## Purpose

This document defines the repo-specific refinement contract for Kris' Lab.

It exists to improve execution quality, polish, and consistency without redefining product direction.

It is:

- a refinement layer
- a craft-improvement lens
- a safeguard against generic AI polish behavior

It is not:

- shell design law
- product direction
- architecture guidance
- a universal premium-UI playbook
- a source of truth above repo docs or code

## Role In The Documentation System

Use this file only after higher-authority sources are already understood.

This file does not decide what Kris' Lab should be. It helps refine how well an already-established direction is executed.

Within the docs system:

- `AGENTS.md` defines agent behavior and implementation discipline
- `docs/shell/*` defines shell law
- local code defines current implementation reality
- this file defines how to refine execution without changing direction

## Authority Order

Use this authority order exactly:

1. explicit task constraints and design directives
2. `AGENTS.md`
3. repo design law such as `docs/shell/*`
4. code reality
5. this taste layer

Practical rule:

- this layer may improve execution quality
- it may not change what the task, shell docs, or repo identity require

## What Good Refinement Means Here

Good refinement in Kris' Lab means:

- improving finish without changing direction
- reducing inconsistency without broad restyling
- tightening hierarchy without adding spectacle
- improving spacing, rhythm, and clarity before adding new visual effects
- preserving strong existing decisions
- keeping shell polish restrained
- keeping piece autonomy intact

Refinement should improve execution quality, not reopen the concept.

## What This Layer Is For

Use this layer to guide:

- shell polish
- local consistency cleanup
- craft upgrades inside an already established direction
- preserve-first visual refinement
- piece-local refinement only when the task is explicitly about that piece

This layer may improve how something is executed.

It may not redefine what the shell or a piece is supposed to be.

Successful shell identity is preserved by default.

Unless a task explicitly authorizes redesign, refinement work must not broadly reopen:

- dock direction
- window direction
- desktop or background direction
- shell material language
- shell interaction character

Refinement may correct inconsistency inside those areas. It may not replace working decisions broadly.

## What This Layer Is Not For

This layer must not be used for:

- creating new design direction
- shell redesign
- generic premium-UI upgrading
- flattening piece autonomy
- replacing the current shell identity with external taste defaults
- broad component abstraction
- introducing louder motion, more glass, or decorative gradients as default polish moves

It must not become:

- shell design law
- a detailed shell style spec
- piece art direction
- a generic frontend style framework
- external skill installation guidance

## When To Use This Layer

Agents may use this layer when:

- the task is explicitly about polish, refinement, craft, or consistency
- shell or piece direction is already established
- the work is narrow and preserve-first
- the likely result can be checked visually
- the change is about execution quality rather than conceptual direction

## When To Ignore This Layer

Agents must ignore this layer when:

- the task is architecture, routing, runtime, data, or bugfix work without visual intent
- the user asks for strict implementation of an existing spec or doc
- the task is defining shell law or product direction
- using this layer would make shell chrome louder than `docs/shell/*` allows
- the task touches piece internals without an explicit piece-level refinement request
- the likely result is generic website polish rather than repo-specific refinement

## Common AI Refinement Failure Modes

AI refinement often fails by:

- making everything more “premium” in the same generic way
- increasing glass, gradients, shadow, or motion without structural need
- replacing local character with polished-but-interchangeable web aesthetics
- overusing cards where spacing or grouping would work better
- changing fonts, radii, or materials too broadly
- making shell chrome louder during polish
- importing asymmetry, motion, or visual drama without compositional reason

Treat these as default failure modes to guard against.

## External Reference Rule

External taste-skill repos or frontend refinement systems may be used as references for:

- audit discipline
- refinement process
- identifying generic AI design habits
- spotting polish debt
- stress-testing consistency

They must not be imported as direct design truth.

Kris' Lab is not:

- a generic SaaS app
- a landing page
- a universal premium-UI exercise

If an external taste heuristic conflicts with the calm shell, piece autonomy, preserve-first refinement, or repo-specific design law, ignore it.

The following external defaults are non-authoritative in this repo unless a local task explicitly calls for them:

- design-variance dials
- motion-intensity baselines
- generic font bans or swap rules
- asymmetry-for-its-own-sake
- broad premium-web patterns
- default cardification
- glass escalation
- motion escalation as a sign of quality

These are optional reference ideas at most. They are not default instructions for Kris' Lab.

## Refinement Batch Rule

Refinement should happen in narrow, legible batches.

Prefer one class of improvement at a time, such as:

- spacing rhythm
- visual noise reduction
- shell chrome consistency
- state clarity
- material cleanup
- alignment and hierarchy repair

Do not combine unrelated polish moves into one broad “improvement pass.”

## Prompting And Delegation Rule

Use this layer only as a final refinement lens, never as the initial product brief.

When prompting or delegating refinement work:

- name what already works
- name the local inconsistency
- constrain the batch tightly
- forbid redesign when the task is polish only
- require visual verification

A good refinement prompt should preserve direction and isolate one local quality problem.

## Safeguards

When using this layer:

- preserve successful shell identity unless the task explicitly says otherwise
- treat louder motion, font churn, gradient escalation, and glass escalation as suspect by default
- do not standardize piece internals unless the task explicitly asks for it
- prefer one narrow refinement batch over broad cleanup
- prefer removing inconsistency over adding personality
- state uncertainty if visual verification is incomplete

If refinement makes the result feel more generic, louder, or less repo-specific, it is moving in the wrong direction.
